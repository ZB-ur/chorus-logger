# Lesson — A helper that must abort the parent script cannot run inside `$(...)` — use a global-var pattern instead

## What happened
The `capture_pane()` helper needs to call `exit 6` when it detects a tmux infrastructure error. Running it inside `$(...)` would mean `exit 6` only exits the subshell, leaving the parent script running with corrupted state. Fix: helper writes its stdout to a global `CAPTURE_OUTPUT` variable and exits the script directly; callers read the variable instead of using command substitution. This pattern was forged from a prior cycle-9 plan-defect lesson.

## Transferable lesson
Shell command substitution `var=$(helper)` runs `helper` in a subshell. Any `exit N` inside that subshell exits the subshell, not the parent — the parent continues with `var=""` and a confused control flow. If your helper needs to terminate the script on a fatal condition, it cannot use the natural "return value via stdout" idiom. Either (a) helper writes to a global variable and exits the parent directly, or (b) caller wraps the command-substitution and checks `$?` explicitly.

## Surface area
Next time you write a shell helper that might need to fail-fatally, ask: "if this exits non-zero inside `$(...)`, does the parent script see that?" If the answer is "no," switch to a global-output-variable pattern.
