# Lesson — Route defensive-check failures through your existing fail-open envelope, not a new error path

## What happened
The get_token JSON branch previously returned whatever sat at `claudeAiOauth.accessToken` — null, empty, numeric. A1 added an isinstance + non-empty assertion at the value site. On failure, get_token returns None — which the existing no-token branch already handles (exit 2, fail-open). Smoke gained two cases: A1a (regression cover for the prior KeyError defense) and A1b (NEW isinstance behavior). The raw-string fallback branch is deliberately left unchecked per a prior cycle's "≤2 assertion sites" budget.

## Transferable lesson
When adding a defensive check to a function, prefer returning the same sentinel the function already uses on its known failure paths — so the check rides the existing envelope rather than inventing a new exit. Also: when adding test cases alongside, label which one is "regression cover" vs "new behavior" so a future reader can tell what was always true from what is new.

## Surface area
Next time you add an isinstance/None-check inside an existing function, find that function's existing "I don't have the thing" return and reuse it, then label each new test case as regression vs. new.
