# Lesson — When adding a new protocol alongside an existing one, acceptance must exercise the cross-mode interaction, not each mode alone

## What happened
v0.6 introduces Mode B (event-driven /loop via `lib/monitor-inbox.sh`) as a sibling of Mode A (cron-based). The acceptance plan deliberately keeps `hello-chorus` MIXED: pinger stays Mode A, ponger migrates to Mode B. The cross-mode delivery (Mode A→Mode B and Mode B→Mode A on the same medium) is the acceptance — not "Mode B works in isolation." Each Mode-B-only test would pass while real deployments mix the two.

## Transferable lesson
Adding a new mode/protocol/transport alongside an existing one means the riskiest behavior is the boundary, not either side. Any acceptance that runs the new mode in isolation will give you green checks while production breaks at the seam. Design the smoke domain so producer and consumer are deliberately on different modes; that's where reality lives. Same for v1→v2 API rollouts, sync→async migrations, single→multi-tenant transitions.

## Surface area
Next time you ship a new mode alongside an old one, write at least one acceptance test where producer and consumer use DIFFERENT modes — the seam is the only test that catches integration failures.
