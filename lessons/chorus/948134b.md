# Lesson — Run logs should carry a machine-parseable footer next to the human prose

## What happened
Twelve per-task run logs were committed, each containing the real stdout from the smoke test plus a strict JSON footer (task #, sha, files touched, smoke command + exit code, spec-violation self-check). A BLOCKED-task-9 log was preserved as historical evidence for a bug fixed three commits later, instead of being deleted.

## Transferable lesson
A run log written only for humans is unsearchable; a JSON dump alone is unreadable. Pair them: free-form prose for reasoning, a fixed-schema footer for grep/jq. And do NOT delete the logs of failed/blocked attempts after a fix lands — the trail from "BLOCKED → root cause → fix commit" is the most valuable evidence you'll have when the bug reappears in a different form.

## Surface area
Next time you write a build/run log, decide the footer schema before the first log, and keep failed-attempt logs in-tree with a 'BLOCKED' prefix rather than deleting them once fixed.
