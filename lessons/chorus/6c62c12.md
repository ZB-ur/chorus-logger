# Lesson — When you need a near-twin script, write a sibling — and apply known corrections at first land, not in retrofit

## What happened
`lib/inject-monitor.sh` is a sibling of v0.5's `lib/inject-loop.sh` (frozen). Rather than parameterize inject-loop or `source` it, a new file was written following the same patterns. All three prior cycle-5b/5c/5d corrections were applied INLINE at first commit (buffer-wide ❯ match, no-literal-middle-dot regex, 180s timeout), so the new sibling has zero "history of the same bugs." The frozen sibling stays immutable.

## Transferable lesson
Two scripts that share a shape but solve different protocols/modes are clearer as siblings than as one parameterized abstraction. Reuse-via-flags pollutes both call sites; the abstraction has to know about both modes' edges. A sibling pays a duplication cost up front but stays readable. Also: when you write the sibling, fold every known-fix from the original into the FIRST commit — don't recreate the bug-history. The bug history is in the original's git log; the sibling should start at the post-fix state.

## Surface area
Next time you're tempted to add a `--mode=B` flag to an existing script, ask if a sibling would be simpler. If you create a sibling, port forward every known fix from the original in the first land — never as separate "now apply X to sibling too" commits.
