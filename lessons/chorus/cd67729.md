# Lesson — Distinguish "missing" from "corrupted" on read; fail loud on write but graceful on read

## What happened
New `read_bookmark` / `write_bookmark` helpers. On read: a missing file returns the epoch `1970-01-01T00:00:00Z` silently (graceful first-run); corrupted content also returns the epoch BUT prints a stderr warning. On write: any non-ISO-8601 input causes exit 1 (fail-loud). The asymmetry is deliberate — readers should keep the system running even with weird state, writers should refuse to perpetuate it.

## Transferable lesson
For state file primitives, missing and corrupted are not the same: missing usually means "system hasn't reached this point yet" and is benign; corrupted means "something wrote garbage here" and is a health signal. Always emit a warning (not an error) when reading corrupted state, even if the function returns the same default. And: reads tolerate, writes refuse — your write helper should reject malformed input so a single buggy caller can't poison every future read.

## Surface area
Next time you write a read/write pair against a state file, give read three return modes (ok / missing-default / corrupted-default-with-warning) and make write fail-loud on schema violations.
