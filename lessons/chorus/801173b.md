# Lesson — Assertion placement determines failure semantics — put it where the existing fail-open envelope catches it

## What happened
A2 checks `five_hour.utilization` and `seven_day.utilization` are correct types inside `fetch_usage`'s success path, BEFORE returning data to main(). On mismatch it returns `{error: 'shape mismatch: ...'}`. The placement is deliberate: putting the assertion later at `maybe_pause` entry would bypass the fail-open envelope (main returns 1, pause-flag UNCHANGED) and could leave the system in a half-paused state.

## Transferable lesson
The location of a runtime check is a semantic choice, not just a code-organization one. The same check placed at the boundary where data enters a function gets caught by the function's existing error envelope; the same check placed inside the consumer bypasses that envelope entirely. Always trace what envelope you want to ride and place the check upstream of it.

## Surface area
Next time you add a shape assertion or invariant check, draw the call path from data-entry to first decision; place the check between them so an existing error envelope handles failure.
