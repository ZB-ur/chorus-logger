# Lesson — Ambiguous spec wording can break downstream runtime assertions — tighten it next to the assertion that depends on it

## What happened
Two non-blocking observations from the prior cycle landed as a polish commit. O2: "touch with timestamp" could be read as `touch -t`, which leaves the file empty and would break a downstream §3.5 C2 ISO-8601 content-shape assertion at runtime. Tightened to "write ts via `printf` not `touch -t`" with explicit reference to the assertion. O3: criterion 9 conflated a pre-approved fallback with design-change failures; split into two clauses.

## Transferable lesson
Spec ambiguity is a runtime hazard whenever a separate assertion downstream depends on the unstated reading. When a reviewer flags "this could be read two ways," don't just clarify — point at the assertion that would break under the wrong reading. The cross-reference makes the tightening surgical and tells the next reader why the wording matters. Also: distinguish "pre-approved fallback" from "design-change failure" in any spec that lists acceptable failure responses; collapsing them invites the wrong cycle of recovery.

## Surface area
Next time a reviewer notes "this wording is ambiguous," check whether any downstream assertion encodes the disambiguation; if so, edit the wording with a reference to that assertion in the same line.
