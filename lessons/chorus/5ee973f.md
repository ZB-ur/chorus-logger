# Lesson — Enumerate known-defect classes in the plan preamble; force every task to declare which one applies

## What happened
The v0.6 plan's preamble enumerates 8 bash defect classes accumulated across prior cycles (CAPTURE_OUTPUT pattern, process-substitution surprise, PATH-shadow mocks, etc.). Each task entry explicitly calls out which class applies and how it is avoided. Conditional logic (Task 11 step 3 only triggers if Task 1 PoC step 5 fails) is pre-approved per the spec so it doesn't re-enter mini-cycle review. Self-review at the plan end verifies every spec section has task coverage.

## Transferable lesson
A "be careful" comment in a plan is dead weight; a checklist of named defect classes is reviewable. Maintain a growing list of failure modes you've seen in this codebase/language/stack and reference them by ID in every task ("this task touches class #3 — uses PATH-shadow per cycle-5"). The list turns intuition into something a reviewer can verify line-by-line.

## Surface area
Next time you write an implementation plan, attach the prior-defect-class list as a numbered preamble and require each task to state which classes it touches plus the mitigation.
