# Lesson — Land the spec in the runtime repo before any code, separate from the methodology repo

## What happened
Imported a 553-line design spec + 1249-line plan into a fresh sibling repo, deliberately not nested inside the methodology repo. Spec enumerated 11 already-decided architecture choices and what v0 explicitly does NOT do.

## Transferable lesson
A runtime repo's first commit should be the frozen design that justifies its existence, not code. Listing what is excluded (no GUI, no schema-first, no cross-device) is as load-bearing as listing what is included — it stops future-you re-litigating settled decisions. Sibling repos beat nesting when methodology and runtime have different release cadences.

## Surface area
Next time you start a new tool/runtime that descends from an existing methodology repo, write the design+exclusions doc as commit #1 in a sibling repo, not a subdirectory of the parent.
