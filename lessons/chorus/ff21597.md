# Lesson — Event subscribers that arm late must emit an explicit catch-up sentinel on startup

## What happened
`monitor-inbox.sh` emits an `inbox-change` sentinel on startup BEFORE entering its poll loop. The reason: messages can arrive while the role's session is down or restarting; if the monitor only fires on mtime-change after arming, those messages stay invisible until the next change. The startup emit guarantees the first wake processes any backlog — the "catch-up framing" finding from a prior cycle.

## Transferable lesson
Any event-driven subscriber that can be "off" while events fire (file watcher, message queue consumer, pubsub client) needs a catch-up step: on startup, emit one synthetic "you may have missed something — go check" event before listening. Without it, the system silently swallows messages between cycles. The catch-up is harmless when there's nothing to catch up — the subscriber re-checks, finds the bookmark up to date, and idles.

## Surface area
Next time you arm an event listener, add a one-line "emit-on-arm" before the listen loop, and make the consumer idempotent so the catch-up emit is a no-op when there's no new state.
