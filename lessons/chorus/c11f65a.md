# Lesson — When you add a config-schema field with related fields, enumerate the full truth table and reject illegal combos at parse time

## What happened
A new `mode: cron | dynamic` field was added to roles.yaml, with `cron:` valid only for Mode A and `monitor_cmd:` valid only for Mode B. The parser enumerated four illegal combos (mode absent + cron absent → error; mode=dynamic + cron present → error; mode=dynamic + monitor_cmd absent → error; etc.) and rejects each with a clear stderr message and `exit 3` at parse time — not when the role tries to run. Also: the per-role loop used `while … done < <(…)` rather than `… | while …` because the latter loses variables across iterations (each pipe stage runs in its own subshell).

## Transferable lesson
A new config field with conditional siblings is a Cartesian product of valid + invalid combinations. Enumerate them in a truth table inside the parser and reject every invalid cell at load time, not at runtime — the runtime error will fire deep inside the wrong code path and confuse the operator. Also: `pipe | while read` loses any variable mutation inside the loop because each pipeline stage runs in a subshell; for cross-iteration state, use process substitution (`while read; do …; done < <(producer)`).

## Surface area
Next time you add a kind/type/mode field to a config, write the truth table of (present + missing) combos with its peers and add an explicit rejection per illegal cell. And: if a `while` loop needs to mutate variables visible after the loop, never pipe into it.
