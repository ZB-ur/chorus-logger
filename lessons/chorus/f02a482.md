# Lesson — Distinguish "infrastructure broken" from "thing not ready yet" with different exit codes

## What happened
Every `tmux capture-pane` call now goes through a `capture_pane()` helper that asserts non-zero exit + checks stderr sentinels (`no such target`, `can't find pane`, `no server running`) and maps them to `exit 6` (tmux infra error) with an attach hint. Stage 1 timeout (probe expired) stays `exit 5`. Operators reading the exit code immediately know whether to retry, attach to tmux, or restart the runtime entirely.

## Transferable lesson
"Probe failed" and "the substrate the probe runs on is broken" look identical at a single bit of failure but demand opposite responses (wait/retry vs. fix infra and restart). Give them different exit codes plus a sentinel-detection helper that wraps every infrastructure call, so the difference cannot be missed. Exit-code-as-routing-key beats parsing stderr at the call site.

## Surface area
Next time you write a probe/health-check, reserve one exit code for "infrastructure error" separate from "probe target not ready" — and centralize the infra-error detection in a single helper.
