# Lesson — Supervisors should be observation-only with an explicit "unknown" verdict; auto-action requires a different commitment

## What happened
`chorus-status.sh` reads runtime state files and outputs one of four verdicts per role: `OK`, `suspect` (only for Mode B — claude+Monitor alive but last-emit aged), `dead`, or `unknown` (role in roles.yaml but no pids entry). Mode A has NO `suspect` state by design — the cycle-6 lesson taught that pane-scrape supervision is unreliable, so the tool is honest about that limit. The tool never auto-kills, auto-restarts, or auto-anything; it observes.

## Transferable lesson
A supervisor that acts has to be right about every signal; a supervisor that observes can be honestly uncertain. The "unknown" verdict — emitted when the supervisor can see the role exists in config but has no PID record — is more valuable than a fabricated guess. Also: if your evidence channel (pane scrape, heartbeat, log tail) has known unreliability, do not invent a "warning" verdict for that channel. Either trust it or surface the limit. Observation-only tools also keep the system safe when the supervisor itself has a bug.

## Surface area
Next time you build a status/health tool, list the verdicts you can emit and the evidence needed for each; if any verdict requires evidence you don't trust, drop it. Resist adding auto-restart until you've watched the observation-only tool for a while.
