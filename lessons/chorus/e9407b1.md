# Lesson — Internalize conditional-approval conditions as explicit tasks with verification methods, not as comments

## What happened
Cycle-5 reviewer issued "APPROVED conditional on M1/M2/Decision-2." The plan didn't bury those in a comment — it added Task 4b for M1 (launchd live-run sanity), made M2 a fail-fast SHA-prefix assertion as Step 2 of the tag task, and split smoke S9.A1 into A1a + A1b per Decision-2. Each step carries an explicit verification-method column (diff/grep/git-op/TDD). Five sequential reviewer cycles (scope → timeout semantics → code placement → pre-spec check → spec audit → plan audit) were tracked by name in the commit.

## Transferable lesson
A reviewer's "approved IF" is a contract, not a suggestion. Conditions written as TODO comments rot; conditions written as numbered tasks with a verification method survive. Also: number your review cycles and quote them by name in the commit body — the audit trail is the proof that approval was real, not assumed.

## Surface area
Next time a review approves your work with conditions, add a task per condition with a verification column before merging — and reference the cycle by name in the commit message.
