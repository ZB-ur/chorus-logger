# Lesson — When a "same bug" recurs, check whether the failure mode is actually different — and write a separate BLOCKED log if it is

## What happened
After the cycle-5b Stage 1 fix landed, the next reboot still showed pinger failing Stage 2 (31s vs 30s --scheduled-timeout). Looked superficially like "task-11 returning." But the post-mortem capture-pane DID contain `Scheduled 7e28aa0f` — it just appeared after the timeout budget, with a `Cogitated for 47s` agent-latency footer. Ponger in the same boot succeeded at 24s. Distinct failure mode: not a detection-location bug, a timing-budget bug. Committed as `BLOCKED-task-11b.md`, separate from BLOCKED-task-11.md.

## Transferable lesson
A symptom that looks identical to a "fixed" bug can have an entirely different root cause. Before deciding "it came back," capture post-mortem evidence (logs, screenshots, pane buffer) and ask whether the *failure mode* matches the old one, not just the *symptom*. Write a new BLOCKED log per distinct cause rather than amending the old one — two short logs that name the cause beat one long log that conflates them.

## Surface area
Next time a "fixed" bug appears to return, gather post-mortem state and check whether the failure mode is the same; if it's a sibling cause, write a new diagnostic with a sibling name (X / Xb / Xc) rather than reopening the original.
