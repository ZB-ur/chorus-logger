# Lesson — When you bump a defensive bound, write an escalation tripwire so the next bump can't quietly happen

## What happened
PoC observed 110s restart latency vs the 90s SCHEDULED_TIMEOUT default. Cycle-5d approved a bump to 180s (≈1.6x margin), matching the v0.5 cycle-5c 30→90s precedent. Critical addition: an S5 tripwire — "if real 2-reboot times approach 180s, escalate to v0.7 redesign, do not bump again." The BLOCKED log documents both PoC findings (Step 5 zombie process tree, Step 6 timing) plus the three approved resolutions (Q1/Q2/Q3).

## Transferable lesson
A timeout/retry/threshold that has been bumped once will be bumped again unless you name the failure mode that would have to be different to justify the next bump. Without an escalation tripwire, every new observation "just barely" crosses the line and quietly re-bumps the bound; six months later your "30 second" budget is 600 seconds and nobody knows why. Document the bump-vs-redesign threshold in the BLOCKED log or code comment.

## Surface area
Next time you bump a defensive constant, add a one-line tripwire: "if observed value reaches X, that is a different failure class — redesign, do not raise this value further."
