# Lesson — A doc that gates execution must enumerate every entry condition AND a 3-step failure response — never just "check these before starting"

## What happened
`code-playbook.md` lists five hard entry conditions for `/bonfire:code` (pipeline_stage=plan, stage-j passed, no pending_reentry, code_ready=true, repo target exists, no unresolved high-impact gaps). Below the conditions, an explicit failure response: "If any gate fails: (1) Do not modify the repo. (2) Log the failure. (3) Report to user with guidance." Three discrete steps in order.

## Transferable lesson
"Check these conditions before proceeding" is incomplete documentation — it tells the actor what to verify but not what to do on failure. The result: partial execution (some checks pass, some fail, runs anyway), silent skipping (logged nowhere), or surprise user dialog (no clear next action). The full pattern is: enumerate conditions, then enumerate failure response as 3+ discrete steps (often: don't touch state, log the failure with detail, report to operator). Each step is a separate sentence so it's hard to miss.

## Surface area
Next time you write a gating reference doc, pair the conditions list with a numbered "if any gate fails" response — three steps minimum (no-op, log, report).
