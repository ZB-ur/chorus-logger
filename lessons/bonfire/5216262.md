# Lesson — Orchestrator skills should ship as a concrete numbered process with literal CLI commands — never abstract guidance; pending-state recovery is step 1

## What happened
`skills/pre/SKILL.md` is the /bonfire:pre orchestrator. Structure: YAML frontmatter (name/description/argument-hint), `<objective>` + `<execution_context>` + `<process>` tagged sections. The process is 17 numbered steps with literal CLI commands and `Agent({ subagent_type: "..." })` invocations. Step 1 checks `pending_reentry` in state.json and routes to recovery BEFORE doing anything else. References are loaded via `@references/...` pointers at startup.

## Transferable lesson
Three habits for orchestrator skills. (1) Numbered steps with literal commands — "execute `tool foo --bar`" beats "use the tool to do foo." (2) Pending-state recovery is step 1, not buried mid-flow — every wake should start by asking "am I resuming an interrupted run?" before any new work. (3) References loaded up-front via @ pointers makes the dependency set explicit and lets the harness preload context. The orchestrator is essentially executable documentation.

## Surface area
Next time you write an orchestrator-style skill/runbook/SOP, write it as numbered steps with literal commands, put "check for pending state" as step 1, and list referenced docs via @ pointers in an execution-context block at the top.
