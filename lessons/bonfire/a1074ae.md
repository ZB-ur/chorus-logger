# Lesson — Templates should mirror the data shape, one section per top-level field; let the renderer rather than the template do the iteration

## What happened
`templates/stage-a.md` renders the approval pack: one section per top-level field — `Reframed Goal`, `Retained Scope`, `Excluded Scope`, `Critical Assumptions`, `Frozen for Code`, `Ambiguity Points`. List fields use `{{#each scope}}- {{.}}{{/each}}` — the template doesn't invent grouping or sorting; it iterates whatever the data provides. Field names in the template match field names in the source JSON.

## Transferable lesson
Templates that invent display structure (sections that don't map to a data field, sorts, groupings, derived counts) become a second source of truth that drifts from the data shape. Map one template section per top-level field; let the renderer iterate arrays as-is. Field names should match data field names so a reader of either side can predict the other. When you need a different shape for display, transform the data BEFORE handing it to the template — never re-shape inside the template.

## Surface area
Next time you write a Markdown/HTML template fed by structured data, lay it out as one heading per top-level data field. Reshape data before rendering, not inside the template.
