# Lesson — When code reads a declarative schema at runtime, write tests that validate the schema itself — schema bugs are silent in renderer tests

## What happened
Six tests target the schema file directly, not any code that reads it: exactly 22 notes (count check), unique ids, unique filenames, `requires` references valid ids (no dangling refs), exactly 9 reentry routes, reentry route targets exist as steps. A duplicated id or a typo'd `requires: "stage-h"` (when the id is `stage-h-review`) would silently produce wrong renders; these tests catch it at the schema line.

## Transferable lesson
Renderer tests verify "given input X, produce output Y" — they don't verify "the schema declaring the structure is internally consistent." A dangling `requires` reference, a duplicated id, or a missing route maps to a runtime silent miss: the renderer just doesn't run that note, or runs it with empty data. Write schema-integrity tests as soon as the schema has ≥2 cross-references (id↔requires, route↔step, filename uniqueness). They fail at the right line.

## Surface area
Next time you commit a JSON/YAML schema with ids and references between them, add 4-6 tests on the schema directly: count, uniqueness, dangling-reference, type-shape. The tests live next to the code that reads it.
