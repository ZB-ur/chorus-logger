# Lesson — Use PID-suffixed tmp + rename for atomic writes; in JSON loaders, distinguish "file absent" (return None) from "real error" (re-raise)

## What happened
`writeAtomic(filePath, data)` writes to `filePath + '.tmp.' + process.pid` then `renameSync` over the target — concurrent invocations of the same tool can't corrupt each other (different PIDs → different temp files). `loadJSON(filePath)` returns `null` on `ENOENT` but re-raises any other error. `exitJSON`/`exitError` enforce a consistent JSON output shape from CLI entry points.

## Transferable lesson
Two paired habits when writing CLI helpers. (1) Atomic write: write to `<target>.tmp.<pid>` then rename — rename is atomic on POSIX. The PID suffix is what makes it safe when two invocations race. (2) In your JSON loader, ENOENT and "parse error / permission denied / EISDIR" mean different things to the caller — collapse them and you lose the ability to handle "first-run, no file yet" as a normal case. Return None/null only for absence; re-raise everything else.

## Surface area
Next time you write a file that might be read concurrently, use a PID-tmp + rename. Next time you write a `loadX(path)` helper, special-case ENOENT and let the rest bubble up.
