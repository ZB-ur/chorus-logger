# Lesson — Declare your pipeline as a schema of (note-id, source, template, requires) and let the renderer be a generic interpreter

## What happened
`schemas/bonfire-v1.json` enumerates every note the pipeline produces with four fields: `id`, `filename` (e.g. `20-b-divergence.md` — number-prefixed for stable ordering), `template`, `source` (e.g. `case.json#stages.divergence` JSON-pointer-style), and `requires` (list of upstream notes). The renderer reads the schema, walks the DAG, and writes each note from its declared source — no hand-coded `if stage == "b"` branches.

## Transferable lesson
A pipeline with N stages tempts you to write N hand-coded renderers, each duplicating the "read JSON → fill template → write file" loop. Declare the stages as data instead: one schema entry per output, with the dependency graph as `requires`. The renderer becomes a generic interpreter; adding a stage is a schema entry, not a code commit. Filename prefixes (`10-`, `20-`, `30-`) give you free lexicographic ordering for navigation.

## Surface area
Next time you have a pipeline producing N markdown notes, write a schema that lists each note's id/filename/template/source/requires, and write ONE renderer that walks the list — never N hand-coded render functions.
