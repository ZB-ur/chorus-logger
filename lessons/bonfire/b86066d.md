# Lesson — Audit logs want JSONL + appendFileSync — no locks, no class, just `appendLog` and `readLog`

## What happened
The logger module is two exported functions and ~32 lines total. `appendLog(filePath, entry)` calls `fs.appendFileSync(filePath, JSON.stringify(entry) + '\n')` — relying on POSIX `O_APPEND` atomicity for single-write-per-call. `readLog(filePath, opts)` parses each line as JSON; returns `[]` on ENOENT or empty file; supports a `{since: timestamp}` filter. Auto-stamps `entry.timestamp` if missing — but the caller can override.

## Transferable lesson
The temptation with "append-only log" is to wrap it in a class with `Logger.create()`, locks, flush intervals. Don't. JSONL (one JSON value per line) is the simplest possible format; `appendFileSync` with `O_APPEND` is atomic per-call on POSIX (writes ≤PIPE_BUF are atomic, single-line entries qualify); the read path is `readFileSync + split('\n') + map(JSON.parse)`. Two functions, zero classes. Auto-stamp the common case, allow override for tests.

## Surface area
Next time you reach for "we need a logging library," check if your needs are met by `appendLog(file, entry)` and `readLog(file)` over JSONL. They usually are.
