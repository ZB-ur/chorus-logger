# Lesson — Plan task names should quote the parameters (counts, dimensions) — "implement X" is uninformative; "implement X with N events, M categories" is auditable

## What happened
Plan 2's task summary names the parameters of each unit: "truth surface (6 events, 8 categories, maturity gates), state machine (step transitions, reentry, cross-skill protocol, run management)." A reader skimming the plan immediately knows the truth surface has 6 event types and 8 categories without opening the spec — and during review, those counts become assertable: if the implementation has 5 event types, something was dropped.

## Transferable lesson
Task names that say "implement <thing>" force readers to open the doc to see what's actually covered. Task names that quote the dimensions — counts, enums, state-machine sizes — turn the plan itself into a checklist. The reviewer can ask "I see 6 events listed; were all 6 implemented?" without context-switching. Same principle for API plans ("REST endpoints (5 verbs, 12 resources)") or migration plans ("backfill 3 columns across 7 tables").

## Surface area
Next time you write a plan task name, quote the numerical dimensions it covers (count of types, enums, states) — make the name self-auditing rather than self-mysterious.
