# Lesson — Ship a "golden" example case under `examples/` — input + expected-output paired, doubles as smoke test AND contributor reference

## What happened
`examples/sample-case/` ships a `case.json` (25 lines of Stage A sample data) plus a `request.txt` (the original 1-line user request that seeded the case). Together they form a "golden" reference case — minimal but real, structurally complete for Stage A, named clearly so anyone exploring the repo knows where to look for "what does a real case look like?"

## Transferable lesson
The `golden` naming pattern is communication: it tells readers "this is the canonical example, do not break it casually." Ship one as early as the data shape stabilizes. Two payoffs: (1) it serves as a smoke test (re-render it, diff vs expected) and (2) it serves as contributor onboarding ("read examples/sample-case/ to see the data shape" beats "read the schema and imagine"). The pairing of `input` (request.txt) + `state` (case.json) is what makes it actionable; either alone is incomplete.

## Surface area
Next time your project produces JSON/structured artifacts, drop `examples/sample-X/` with one minimal-but-complete instance + the input that created it; commit both early so contributors don't have to imagine the shape.
