# Lesson — When you discover a spec drift mid-implementation, do the full alignment in ONE commit — rename + restructure + API change + tests all together

## What happened
A spec-vs-code drift was caught: code used `statement` but spec said `content`; snapshot was an array but spec said object-map keyed by id; supersede API was `old_id/new_id` but spec said `supersedes/id`. ONE commit did all of: rename `statement → content` everywhere, change snapshot from `[...]` to `{ id1: {...}, id2: {...} }`, restructure supersede signature, add `version/replayed_at/event_count/by_status/by_category` to snapshot, fix every test file in lockstep. The diff is large; the intermediate state never exists.

## Transferable lesson
Spec-drift alignment is one of the few times "big commit" is correct. If you rename `statement → content` in five files across three commits, the middle commit has callers that reference both names — and any code that reads BOTH the snapshot AND the history will see incoherent data. Do the full sweep in one commit: rename, restructure, API change, test updates. Also: object-map keyed by id beats array-of-objects when reads are id-keyed lookups; replay rebuilds the map once, callers get O(1) reads forever.

## Surface area
Next time you find code drifted from spec on multiple dimensions, prep a single commit that fixes ALL the drift dimensions at once — never half-rename across a series.
