# Lesson — Separate the durable "truth surface" from the rendered/visible UI in a multi-stage pipeline

## What happened
The bonfire design spec for an Evolution Constraint Planner reimplementation centered on a "truth surface dual-layer system": one layer is the JSON state that downstream stages and agents read/write; the other is the human-facing markdown rendered from it. The pipeline state machine, agent delta protocol, and real-time dual-write rendering all key off the JSON layer, with the markdown layer treated as derived view.

## Transferable lesson
In a multi-stage pipeline where each stage mutates state and humans need to follow along, two design errors are common: (a) writing only markdown — downstream stages have to parse human text, breaking any change to wording, and (b) writing only JSON — humans lose the running view. The right shape is a dual layer: JSON is the durable truth, markdown is rendered from it on every write. Agents propose deltas to the JSON; the renderer produces a fresh markdown view each time. Both files exist; only one is authoritative.

## Surface area
Next time you design a multi-stage workflow with both machines and humans in the loop, separate "what the next stage reads" (structured JSON) from "what the human sees" (rendered markdown), with the renderer running on every JSON write.
