# Lesson — Implementation agents need an explicit CAN/CANNOT decide list and a forbidden-context list — without them, the agent invents product meaning under pressure

## What happened
`bonfire-coder` ships two adjacent lists. CAN decide: helper decomposition, import wiring, local variable names, package lockfile updates, tiny file placement. CANNOT decide: user goal meaning, frozen product semantics, validation meaning, success criteria, dependency behavior, visible UI choices the handoff phrases as alternatives. Ambiguity-handling rule: "do NOT invent the answer. Do NOT search for the answer in bundle/ markdown or plan/ delta files" — explicit forbidden-context list.

## Transferable lesson
An implementation agent given a "frozen" handoff will, under any ambiguity, search wider context (other plan docs, the bundle, git history) for "the answer." Those wider sources contain in-progress reasoning, not frozen decisions — so the agent silently re-derives product meaning from non-frozen sources. The fix: ship a CAN/CANNOT decide list AND a forbidden-context list ("do not search X, Y, Z to fill gaps"). Force ambiguity to surface as an explicit "ambiguity flag" in the output rather than as a quiet guess.

## Surface area
Next time you write a "frozen handoff → implementation" agent prompt, ship paired lists: what the agent MAY decide (low-impact engineering only) and what it MAY NOT search to fill gaps (other plan docs, deltas, history).
