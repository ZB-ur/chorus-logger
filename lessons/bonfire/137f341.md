# Lesson — When an agent gets Write permission, scope it to ONE specific file with a structured schema; enumerate the concrete things that would force rejection

## What happened
`bonfire-h-review` gets Write permission for exactly one file: `.bonfire/plan/h-review-verdict.json` — not a directory, not "plan/*". Rejection criteria are enumerated by category of what the coder would otherwise have to invent: product meaning (what user wants), validation meaning (what counts as correct), state behavior (how data transforms), dependency behavior (what happens when deps fail). "If approving with conditions, conditions must be specific and actionable."

## Transferable lesson
Two paired rules for write-authorized agents. (1) Narrow the write target to one file with a known schema — never grant "write to plan/" or "write anywhere under .bonfire/". The narrow allowlist makes the agent's contract reviewable. (2) Enumerate the categories of "must not invent" that force rejection (product/validation/state/dependency) instead of "is it good enough?" Vague criteria produce vague rejections; categorical criteria produce actionable ones.

## Surface area
Next time you give an LLM agent write permission, name the exact file path (not a directory glob), and enumerate concrete-category rejection criteria so the agent's verdict logic is auditable.
