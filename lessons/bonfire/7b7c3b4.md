# Lesson — When two state files split concerns, refuse to duplicate the same fact in both — pick one as authoritative and reference it from the other

## What happened
`case.json` had an `artifacts.compile_output` field that duplicated information already expressed in `state.json` as `steps.stage-j.status == "passed"`. The duplication was removed; case.json now holds only stage narrative data, while state.json owns step completion status. Two files, two concerns, zero duplication.

## Transferable lesson
When you split system state across multiple files for clarity (narrative vs status, config vs runtime, source-of-truth vs cache), the immediate temptation is to convenience-duplicate a field "in case the reader of file A needs it without opening file B." This duplication is a guaranteed future drift bug. Pick one file as authoritative per fact and require the consumer to consult both files when needed; the inconvenience is cheaper than the drift.

## Surface area
Next time you add a field that already exists in a sibling state file, don't — link to the authoritative location and require the consumer to read it directly.
