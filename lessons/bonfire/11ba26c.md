# Lesson — Ship file-layout docs as a `file → source` table; diagnosis-frame docs need an explicit "when NOT to apply this" guard

## What happened
`obsidian-layout.md` is dominated by a table: every bundle file (00-overview.md, 05-constraint-ledger.md, … 99-final-handoff.md) paired with its source (case.json, truth-surface snapshot, plan/...). No prose describing "the layout" — just the table. `diagnosis-and-observability.md` opens with "It is NOT a default lens for every product-planning request" and lists when to apply it (bugs/regressions/failures) explicitly.

## Transferable lesson
Two anti-patterns this commit corrects. (1) Describing a file layout in prose ("the overview note comes from case.json, the constraint ledger snapshot is rendered from..."). A table beats prose for layouts — readers grep, scan, and edit it; prose accumulates exceptions and rots. (2) A reference doc framed around a powerful lens (diagnosis, security review, performance audit) needs explicit "when NOT to apply" guidance, or readers will force-fit every problem into that lens. The "when not" section should come BEFORE the methodology.

## Surface area
Next time you document a multi-file layout, lead with a `file → source` table. Next time you write a lens-style methodology doc, open with the "when NOT to apply" guard.
