# Lesson — Structural validators should return `{present, missing, errors}` — not pass/fail; operators need to see which specific pieces are or aren't there

## What happened
`validateBundle(root)` walks `schema.notes`, skips `{run_id}`-templated paths, splits combined-source paths on `+`, and returns `{present: [...], missing: [...], errors: [...]}` listing exactly which notes were found and which weren't. `validateHandoff(compileOutput)` accumulates an `errors` array with specific messages (`"Missing required handoff field: implementation_units"`, `"handoff.code_ready is not true"`) — never a bare boolean.

## Transferable lesson
A validator that returns a boolean `valid` forces the caller into "OK or rerun." A validator that returns categorized results (present/missing for structural checks; errors for soft failures) lets the operator see exactly what's wrong without re-running. Same with bundle/manifest/migration validators — "missing 3 of 12 expected files" is more actionable than "invalid." Templated paths (`{run_id}`, `{stage}`) need explicit skip logic before structural existence checks — otherwise you false-fail on path patterns that aren't supposed to be literal files.

## Surface area
Next time you write a validator that checks "are all expected files present," return categorized lists (`{present, missing, errors}`) and explicitly skip path patterns containing `{...}` placeholders before stat-checking.
