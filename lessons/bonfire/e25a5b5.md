# Lesson — A handoff between planning and execution needs an explicit freeze-surface checklist — "code-ready" without enumeration is opinion

## What happened
`handoff-quality-bar.md` defines code-readiness as: "the next coder can implement without inventing high-impact meaning." Then it enumerates the freeze surface as a literal checklist — `repo_grounding`, `frozen_product_decisions`, `domain_model`, `data_contract`, `ui_contract`, `function_contracts`, `file_plan`, `implementation_units`, `verification_commands`, `browser_checks`, `acceptance_checks`. Each implementation unit answers: what/why/files/functions/deps/verification/done-meaning. Web-app handoffs add UI-specific items: state ownership, persistence model, failure-state copy.

## Transferable lesson
"This handoff is ready" without an enumerated freeze surface is opinion that varies by author. Define code-ready as a literal checklist: every category of decision that, if left open, forces the implementer to invent product meaning. Web apps need UI-specific items beyond backend handoffs (state ownership, copy, persistence model). The checklist also surfaces what was deliberately deferred — "we did not freeze X because the user doesn't care" beats "we forgot to think about X."

## Surface area
Next time you hand off a plan to an implementer (yourself in 2 weeks, an LLM, a contractor), write the freeze-surface checklist and either fill each item or explicitly note "deferred — implementer decides."
