# Lesson — `init` writes every expected state file with a fully-keyed schema and null placeholders, and refuses to overwrite an existing workspace

## What happened
`init` creates `.bonfire/` plus seven subdirs (`truth-surface`, `plan`, `bundle`, `runs`, `logs`, `archive`). Then it writes `state.json` and `case.json` with FULL schemas — `case.json` enumerates all 9 stages (`preprocess`, `divergence`, …, `compile_for_code`) with `null` placeholders; `state.json` includes `reentry.depth/max_depth/history`, `approval.*`, `runs.*`. If `.bonfire/` already exists, it refuses with `"Archive the current case first"` rather than overwriting.

## Transferable lesson
Two paired habits for `init`-style commands. (1) Write every expected file with its full schema and `null`/empty-list placeholders for unfilled fields — consumers can read `state.steps['stage-b']` without checking "does this key exist," because init wrote `null` for every key the schema declares. (2) Refuse to clobber existing state — `init` over a populated workspace silently destroys an in-progress case. Force the user to archive first; the friction is the feature.

## Surface area
Next time you write a `bootstrap`/`init` command, write the full state schema with `null` placeholders for all future fields, and refuse to run when the workspace already exists.
