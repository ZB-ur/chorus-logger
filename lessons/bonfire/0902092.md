# Lesson — When you need a templating engine for ~3 patterns, write 50 lines of regex substitution — don't pull in Handlebars/Mustache

## What happened
The renderer supports exactly three template patterns: `{{field}}` (substitution), `{{#each array}}...{{/each}}` (iteration), and `{{.}}` (current primitive in array iteration). Implemented in ~30 lines using `String.replace(regex, fn)` — no external dependency. Each unknown field becomes `''` rather than throwing, so missing data doesn't crash the renderer.

## Transferable lesson
Pulling in Handlebars/Mustache/Jinja for a project that needs three substitution patterns trades 50 lines of obvious regex for a transitive-dependency tree, a learning curve, and a behavior surface that exceeds your needs. When your templating needs fit on one screen, a custom engine is simpler. The trade flips when you need partials, conditionals, helpers, or escaping rules — at that point, take the dependency, but only at that point.

## Surface area
Next time you think "we need a templating engine," count the patterns. If ≤3, write the regex-substitution function yourself; if you need partials/conditionals/escaping/helpers, take the dependency.
