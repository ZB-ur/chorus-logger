# Lesson — Runs templates turn machine output into human-readable narrative — render exit codes + summarized output, not raw logs

## What happened
The runs templates render automation evidence as readable narratives. `verification.md` shows each verification command with `Exit code: {{exit_code}}` + `Output: {{output_summary}}` — summarized, not raw. `reentry.md` captures conflict-type/from-unit/target-stage/reason. `code-run.md`/`achieve.md` capture per-run narratives. The bundle becomes a graph of code-run + verification + reentry + achieve notes per run, linked by run-id.

## Transferable lesson
When automation produces output that humans will later read for context (post-mortems, "what happened during this run?"), don't paste raw logs into a Markdown bundle — they're noise. Render a narrative template: command + exit code + summarized output (first/last few lines, or a producer-chosen summary line). Raw output stays in a sibling log file for forensic dives; the narrative stays scan-able. Same idea for CI run reports, deploy logs, migration runs.

## Surface area
Next time you template the output of an automated run (CI, deploy, migration, smoke), render `{ command, exit_code, output_summary }` per step — not raw stdout. Keep raw logs in a sibling file for the rare deep dive.
