# Lesson — Agent definitions go in tagged sections (`<role>`, `<rules>`, `<input>`, `<output>`) with the output as a literal JSON example

## What happened
Three Stage A agent .md files (reality-checker, intent-extractor, blind-spot-scout) share a structure: YAML frontmatter (name/description/tools), then `<role>` / `<rules>` / `<input>` / `<output>` tagged sections. Output is given as a literal JSON example with sample fields (confirmed_facts[].evidence, dubious_claims[].reason, etc.). Reality-checker's first rule: "Search the actual repo before accepting any user claim as fact."

## Transferable lesson
Agent prompts that mix role description, rules, input shape, and output shape into prose are hard to maintain (and easy for the LLM to gloss over). Tagged sections give the model clear semantic landmarks — `<rules>` is what it MUST follow, `<output>` is what it MUST produce. Putting the output shape as a literal JSON example with placeholder values makes the contract checkable: the parent skill can `JSON.parse` and assert known keys, and a future agent revision can be diffed at the example.

## Surface area
Next time you write an LLM agent prompt, split it into `<role>`, `<rules>`, `<input>`, `<output>` tagged sections and put the output as a literal JSON example, not a prose schema description.
