# Lesson — When rendering a templated output that must match a spec, write the full expected string literally in the test — don't reuse the template constants

## What happened
`render_card(project)` produces a 6-line Markdown card. The two main tests write out the FULL expected string as a multi-line literal: `"# foo\n**Stack**: Python\n**Last activity**: 2026-05-01 (17 days ago)\n**Status**: active\n**One-liner**: a thing\n**Suggestion**: continue"`. The all-None case also has its full expected string. A small `_or_unknown` helper covers the None→"unknown" substitution.

## Transferable lesson
Tests that reuse the template's own format strings to construct "expected" assertions test that the template is self-consistent — they cannot catch a typo or wrong-key in the template itself. Write the expected string literally, character by character. The cost is duplication; the benefit is that the test is a frozen specification independent of the code. Two endpoints (full-real-values + all-defaults) usually cover the lattice corners enough; intermediate cases can use the smaller `assertIn` form for cross-cutting properties.

## Surface area
Next time you write a renderer/template/formatter test, write the expected output as a hardcoded literal — never as a format-string call that reuses the template's own substitutions.
