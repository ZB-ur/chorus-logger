# Lesson — When a conditional uses truthiness, test the empty-string case explicitly — the test pins the contract against future "is not None" refactors

## What happened
`derive_oneliner` uses `if readme_line:` (truthiness) rather than `if readme_line is not None:`. The test suite explicitly includes `test_empty_readme_falls_through_to_commit`: an empty `""` should fall through to the commit msg. Without that test, a future refactor to `if readme_line is not None:` would break the contract (empty-string readme would suddenly win over a real commit-msg fallback) and no test would notice.

## Transferable lesson
Truthiness and not-None are different contracts in Python: `""`, `0`, `[]`, `False` are all falsy but not-None. When your code uses truthiness, write a test that exercises a falsy-but-not-None value — that test is what pins the contract. Without it, the "obvious cleanup" PR breaks behavior silently. Same principle for JS (`if (x)` vs `if (x !== undefined)`), Ruby (`if x` vs `if !x.nil?`), etc.

## Surface area
Next time you write `if value:` or `unless value:`, add at least one test using a falsy-but-non-None value (`""`, `0`, `[]`, `False`) that exercises the falsy branch — pin the truthiness contract before someone "improves" the check.
