# Lesson — Order a TDD plan from inside-out: pure functions → file-touching helpers → orchestration → CLI/shim → smoke

## What happened
The 12-task plan went: dataclass scaffold → pure functions (status, suggestion, oneliner) → file-touching helpers (stack, readme, mtime, git) → render → scan_root orchestration → CLI + shim → smoke.sh. Each task followed a strict TDD micro-cycle: failing test → run → minimal impl → run → commit. Pure functions came first; CLI came near-last.

## Transferable lesson
TDD plan ordering matters more than the TDD commitment itself. Start with the innermost pure functions you can write and test without touching the filesystem, then add file-touching helpers (still testable in isolation via tmp_path), then orchestration that wires them, then the CLI/shim that's the entry point. If you write the CLI first, you'll either have a slow black-box integration test or retrofit testability later — both worse than writing the pure parts first.

## Surface area
Next time you draft an implementation plan, name your innermost pure functions in Task 2-3 and your CLI entry point in Task 10+. Resist the urge to "get something running end-to-end" before the pieces are individually proven.
