# Lesson — Carry-forward type-discipline fixes into the next commit; add a final `raise` to defend against future Literal extensions

## What happened
A `Status = Literal["active", "dormant", "abandoned", "unknown"]` type alias was added and applied as `derive_status`'s return type — a "carry-forward from Task 2 quality review" rather than reopening Task 2. The new `derive_suggestion(status: Status)` ends with an explicit `raise ValueError(f"unknown status: {status}")` after handling all four current values. Each branch is tested; the `raise` is also tested.

## Transferable lesson
Two paired habits. (1) When a quality review surfaces a type-discipline gap (missing alias, missing annotation), apply the fix in the NEXT commit you're already writing rather than amending the original — the git history stays clean and the review's authority is preserved. (2) When you exhaustively match on a Literal, end with a `raise` that names the unexpected value — Python won't tell you if the Literal grows to 5 values; your runtime exception will, immediately, with the exact bad value in the message.

## Surface area
Next time a code review adds a type alias, fold it into your next commit as a carry-forward. Next time you exhaustively match a Literal, end with `raise ValueError(f"unknown <name>: {value}")` even when it "can't happen."
