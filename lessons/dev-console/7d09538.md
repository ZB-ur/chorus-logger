# Lesson — When parsing tool output for sentinel patterns, anchor both ends of the regex; free-text fields will eventually contain your sentinel word

## What happened
`scripts/smoke.sh` parses dev-console output to count "passing" cards (≤1 unknown of 5 core fields). The awk regex is anchored: `/^\*\*Stack\*\*: unknown$/` rather than `**Stack**:.*unknown.*`. The comment explicitly notes "anchored to 'unknown$' — no `.*` — so user free-text in One-liner can't false-positive (per design doc smoke.sh fix)." Without the anchor, a project whose oneliner reads `"some unknown thing"` would have counted as a Status unknown.

## Transferable lesson
Smoke / health / grep-based checks that read tool output need both-end anchors on their regexes. The day someone's README says "this is the unknown future" or their commit message says "TODO: handle unknown case", your smoke counts it as a real `unknown` and silently lies. Anchor `^` and `$`, escape the field separator literally, and avoid `.*` unless you can prove the wildcard span is structurally bounded.

## Surface area
Next time you grep/awk tool output for status sentinels, anchor both ends of the regex and convince yourself no user-controlled string could match the anchored pattern.
