# Lesson — Assert on the specific exception type and all fields of a dataclass; broad assertions hide real failures

## What happened
Quality review tightened two test weaknesses in the dataclass scaffold: (1) `assertRaises(Exception)` was replaced with `assertRaises(AttributeError)` — because the broad form passes if your code raises *any* exception, including `NameError` from a typo. (2) The construction test asserted only 3 of the dataclass's 7 fields; now it asserts all 7 — a missing assertion lets a field swap or typo slip through.

## Transferable lesson
"It raised something" is rarely what you mean — name the specific exception class. And when you construct a value type with N fields, assert on all N: a partial assertion gives a false sense of coverage, because the field you didn't check is the one that breaks. The cost is a few extra lines; the payoff is that the test fails for the *real* reason when it fails.

## Surface area
Next time you write `assertRaises(Exception)` or check only "a few representative fields" of a value type, replace the exception with the narrowest class and assert every field — the diff is small, the safety is large.
