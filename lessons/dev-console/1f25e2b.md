# Lesson — Separate the CLI module from its entry-point shim; the shim is plumbing, the CLI module stays testable as Python

## What happened
The dev-console CLI lives in `cli.py` (argparse + `main()`). A separate `./dev-console` executable shim (≈11 lines: shebang + sys.path tweak + `from cli import main` + `sys.exit(main())`) is what users actually run. The split lets `cli.main()` be imported and tested directly by tests, while the shim handles the "make it runnable from any cwd" plumbing.

## Transferable lesson
A common anti-pattern: stuff argparse setup, business logic, and the `if __name__ == "__main__"` block into a single executable file with no `.py` extension — now you can't import it from tests without monkeypatching `__main__`. The clean form: a `cli.py` module with a `main(argv: list[str] | None = None) -> int` function, plus a tiny shim that adjusts sys.path and calls `cli.main()`. Tests import `cli.main` directly; users invoke the shim.

## Surface area
Next time you write a CLI entry point, name the script `cli.py` and write a one-line shim that imports + calls it — don't merge them into a single executable.
