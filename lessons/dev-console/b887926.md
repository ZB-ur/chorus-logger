# Lesson — When a tool has shared vs local config files, commit only shared and gitignore local — and explain the convention in the commit

## What happened
Claude Code uses two settings files: `settings.json` (team-shared) and `settings.local.json` (user-local, gitignored by default). The commit added `.claude/settings.json` with `superpowers` enabled so a fresh clone gets the plugin enabled by default, while user-local overrides stay out of git. The commit message states the convention explicitly: "settings.local.json is user-local and gitignored; this is the team-shared default."

## Transferable lesson
Many tools split config into shared/local pairs: claude (settings.json + settings.local.json), VS Code (workspace + user), IntelliJ (.idea shared vs workspace.xml local), ESLint, direnv. Pick one rule per tool and document it in the commit: shared = committed default that every clone gets; local = gitignored user override. Without the explanation in the commit, a future contributor will "fix" the gitignore "leak" and break onboarding.

## Surface area
Next time you commit a shared config file with a sibling local-override file, name both files in the commit message and explain which one is committed and which is gitignored, and why.
