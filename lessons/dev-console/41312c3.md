# Lesson — Capture spec wording corrections in the initial commit message itself; future readers see WHY the wording reads as it does

## What happened
The dev-console init commit didn't just land SPEC.md / README.md / design doc / .gitignore — its message explicitly called out the spec wording correction: `"按优先级顺序 concat" — no "first match" language`. The choice of Python stdlib, 3-file layout, tmp_path-based tests, and `smoke.sh` as Done Condition were also captured up-front in the design doc, not deferred to comments inside the code.

## Transferable lesson
A spec correction that landed before code is invisible to anyone reading the repo unless the commit message says so. Future readers see "concat by priority" in the doc and don't know they're benefitting from a deliberate choice over "first match." Quote the rejected wording in the init commit so the choice is auditable. Same idea for stack choices: a one-line "we picked X because not Y" in the init commit is worth a paragraph of later rationale-mining.

## Surface area
Next time you commit a new project's spec/init, quote in the commit message any wording you deliberately rejected ("we wrote X, not Y, because Z") so the choice survives beyond your own memory.
