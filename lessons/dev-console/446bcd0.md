# Lesson — Bound scans of user-authored files, pick lenient encoding, and pin the literal spec ("# H1 yes, ## H2 no")

## What happened
`extract_oneliner` reads README.md/README/README.txt (fallback chain). Three defenses: (1) `_MAX_README_LINES_SCANNED = 50` cap — README.txt might be megabytes; (2) `encoding="utf-8", errors="replace"` — never crash on byte-soup; (3) explicit "skip `# ` only" — a test (`test_h2_not_skipped`) pins the literal spec by asserting `## H2` is RETURNED, not skipped, even though most humans would expect H2-skip too.

## Transferable lesson
Three habits when extracting from user-authored text. (1) Cap the scan — set a line/byte budget so a misshapen file can't hang you. (2) Choose encoding leniency at read time (`errors="replace"`) — unrenderable bytes shouldn't be an exception, just substitution. (3) When the spec is deliberately literal ("skip H1 only"), write a test that exercises the rejected interpretation (H2 NOT skipped) — that test is what stops a future "improvement" from quietly broadening the rule.

## Surface area
Next time you parse user-authored Markdown/YAML/whatever, set a line cap, use `errors="replace"`, and write at least one test for what your code DOESN'T do (so future broadeners get caught).
