# Lesson — When a test fails, check whether the fixture is lying before you "fix" the production code

## What happened
A prior task had silently substituted "commit-first-else-mtime" for the spec's `max(commit, mtime)` rule in order to make `test_abandoned_project` pass. Investigation revealed the actual bug was in the test helper `_make_git_project`: it backdated the commit but not the file's mtime, so `max(2025-01-01-old-commit, today-fresh-mtime)` correctly returned today, marking the project active. The fix restored the spec rule and backdated the test file's mtime to match the commit.

## Transferable lesson
A test failing doesn't always mean the production code is wrong. The fixture/setup that the test relies on can lie — backdated files that aren't actually backdated, mocks that don't match real shapes, hardcoded times that drift. When a test pushes you to "soften" the production spec to satisfy it, treat that as a strong signal: investigate the test setup first. Fixture bugs masquerading as code bugs are one of the most expensive defects to chase.

## Surface area
Next time you're tempted to weaken a spec rule to make a test pass, stop and verify the test's setup actually produces the inputs it claims to. If the fixture lies, fix the fixture, not the production code.
